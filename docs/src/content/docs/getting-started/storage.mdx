---
title: Match Lidarr
description: Mount Aurral like Lidarr. Same media root, same path view.
---

Aurral is a **Lidarr companion**. Storage follows Lidarr, not a separate Aurral layout.

If Lidarr already downloads and your player already sees Lidarr's library, mount that same media root into Aurral. The filesystem side is done.

## The rule

Mount the same media volume Lidarr uses, at the same container path. Most stacks use `/data:/data`.

```yaml
volumes:
  - /data:/data
  - ./config:/config
```

`/config` is Aurral-only (database, settings, secrets). Do not share it with Lidarr.

For path planning, hardlinks, and permissions, use the Servarr guides:

- [Servarr Docker Guide](https://wiki.servarr.com/docker-guide): one common volume, consistent paths
- [Lidarr Docker notes](https://lidarr.audio/#downloads-v1-docker): same `/data` advice for Lidarr images

## After the mount

1. In **Settings → Download Clients → Downloads Folder → Path**, set a folder under that root (for example `/data/downloads/aurral`).
2. Connect Lidarr and run **Test library access**.
3. If you use a player, add Aurral's downloads path as a music folder (Navidrome) or let Aurral register it (Plex). Scanning Lidarr's library alone does not include Aurral's output.

yt-dlp writes straight into that downloads path. If Lidarr already shares `/data` with slskd or Usenet, Aurral sees those completed folders too once you connect the clients in Settings.

## Example layout

This is the usual Servarr-style tree. Yours can differ as long as every container sees the same paths.

```
/data/
├── music/                 # Lidarr root folder
└── downloads/
    ├── slskd/complete/    # slskd finished downloads (if used)
    ├── nzbget/completed/  # optional Usenet completed
    └── aurral/            # Aurral Downloads Folder path
        └── aurral-weekly-flow/
```

| Service        | Mount            | Notes                                               |
| -------------- | ---------------- | --------------------------------------------------- |
| Lidarr         | `/data:/data`    | Root folder e.g. `/data/music`                      |
| Aurral         | `/data:/data`    | Same mount; set Downloads Folder in the UI          |
| slskd          | `/data:/data`    | Same mount Lidarr already uses                      |
| SABnzbd/NZBGet | `/data:/data`    | Same mount Lidarr already uses                      |
| Navidrome      | `/data:/data:ro` | Scan Lidarr library **and** Aurral's downloads path |
| Plex           | `/data:/data`    | Reads Aurral's `aurral-weekly-flow` tree            |

App state stays in `./config:/config` and is not shared.

## Windows and mixed Docker setups

When Lidarr or a download client runs on Windows and Aurral runs in Docker, reported paths like `N:\Music\...` are not readable inside the container.

Mount the Windows folder into Aurral and add a mapping under **Settings → Download Clients → Remote Path Mappings**:

```yaml
services:
  aurral:
    volumes:
      - N:/ServerFolders/Music:/music
      - ./config:/config
```

Example mapping: remote `N:/ServerFolders/Music` to local `/music`. You can also set `PATH_MAPPINGS=N:/ServerFolders/Music|/music` in compose (applied at runtime; edit in the UI afterward).

Then run **Settings → Lidarr → Test library access**.

:::note
Compose `PATH_MAPPINGS` do not appear as env fields in the UI. Use **Settings → Download Clients → Remote Path Mappings** to view or edit them.
:::

## When the player sees different paths

If Navidrome's path view differs from Aurral's, use **Use Navidrome paths in M3U files** under **Settings → Playback → Navidrome**. For Plex, set **Plex downloads path** under **Settings → Playback → Plex**.

See [Navidrome](/integrations/navidrome/) and [Plex](/integrations/plex/).
