title: Shared storage
description: Mount the same media root in Aurral, Lidarr, slskd, SABnzbd/NZBGet, Navidrome, and Plex.
---

Flows, download handoff, Lidarr file reuse, Navidrome playback, and Plex library scanning all depend on **the same host folder mounted at the same container path** across services.

Docker only exposes mounted paths. If Lidarr stores a file at `/data/music/...` but Aurral does not have `/data/music` mounted, library checks and reuse fail even when the API connection works.

Every service mounts the same media root at `/data` inside the container.

```
/data/
├── music/                              # Lidarr root folder
└── downloads/
    ├── slskd/
    │   └── complete/                   # slskd finished downloads
    ├── nzbget/                         # optional NZBGet completed downloads
    └── aurral/                         # Aurral DOWNLOAD_FOLDER
        └── aurral-weekly-flow/           # created by Aurral
```

## What each service needs

| Service   | Mount            | Internal path                                   |
| --------- | ---------------- | ----------------------------------------------- |
| Lidarr    | `/data:/data`    | Root folder `/data/music`                       |
| slskd     | `/data:/data`    | Downloads `/data/downloads/slskd/complete`      |
| NZBGet    | `/data:/data`    | Completed `/data/downloads/nzbget/completed`    |
| SABnzbd   | `/data:/data`    | Completed per SABnzbd config                    |
| Aurral    | `/data:/data`    | `DOWNLOAD_FOLDER=/data/downloads/aurral` (yt-dlp needs no extra mount) |
| Navidrome | `/data:/data:ro` | Scan `/data/music` and `/data/downloads/aurral` |
| Plex      | `/data:/data`    | Read `/data/downloads/aurral/aurral-weekly-flow`|

Aurral app state stays in `./config:/config` and does not need to be shared.

## Example compose file

```yaml
services:
  aurral:
    environment:
      - PUID=1000
      - PGID=1000
    volumes:
      - /data:/data
      - ./config:/config

  lidarr:
    volumes:
      - /data:/data
      - ./config/lidarr:/config

  slskd:
    volumes:
      - /data:/data
      - ./config/slskd:/app

  navidrome:
    environment:
      - ND_SCANNER_PURGEMISSING=always
      - ND_DATA=/config
    volumes:
      - /data:/data:ro
      - ./data/navidrome:/config
```

After startup:

1. In Lidarr, set the root folder to `/data/music`.
2. In slskd, set downloads to `/data/downloads/slskd/complete`.
3. In Aurral onboarding or Settings, set `DOWNLOAD_FOLDER` to `/data/downloads/aurral`.
4. In Navidrome, add `/data/music` and `/data/downloads/aurral` as music folders.

:::caution
Mounting only `/data/music` is enough for Lidarr API connectivity, but Aurral cannot see `/data/downloads/...` unless that tree is mounted too. Mount the parent shared root.
:::

## Windows and mixed Docker setups

When Lidarr or slskd run natively on Windows, they report host paths like `N:\Music\...` that Aurral in Docker cannot open directly.

Fix this by mounting the Windows folder into Aurral and adding `PATH_MAPPINGS`:

```yaml
services:
  aurral:
    environment:
      - DOWNLOAD_FOLDER=/music/Aurral
      - PATH_MAPPINGS=N:/ServerFolders/Music|/music
    volumes:
      - N:/ServerFolders/Music:/music
      - ./config:/config
```

`PATH_MAPPINGS` uses `remote|local` pairs. For multiple mappings, separate with `;`. Scope to a source with `source|remote|local` — for example `lidarr|N:/ServerFolders/Music|/music`.

After adding mappings, run **Settings → Lidarr → Test library access** to verify Aurral can read the reported paths.

:::note
`PATH_MAPPINGS` from compose are applied at runtime but do not appear in the UI. Use **Settings → System → Remote Path Mappings** to view or edit them in the app.
:::

## When Navidrome or Plex see different paths

If Navidrome runs with a different filesystem view than Aurral, use **Use Navidrome paths in M3U files** in **Settings → Playback → Navidrome**. For Plex, set the Plex downloads path in **Settings → Playback → Plex**.

See [Navidrome](/integrations/navidrome/) and [Plex](/integrations/plex/) for details.
